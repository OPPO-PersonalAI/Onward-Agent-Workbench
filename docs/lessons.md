# 经验教训

## 2026-03-26: 多终端刷屏性能优化 — 端到端压力测试

### 1. 测试方式要模拟真实用户场景，不要只测系统指标
- **问题**：最初只测了终端输出的 fps 和 xterm.write 耗时，声称"性能完全没问题"（fps=32，writeMax=0.4ms），但用户实际打字时明显感到卡顿。
- **原因**：混淆了"终端输出性能"和"用户输入性能"。用户的痛点是在终端疯狂刷屏时 Prompt 输入区的响应速度，而不是终端本身渲染是否流畅。测试指标选错了。
- **教训**：性能测试必须从用户体验出发设计指标。先问"用户在做什么操作时感到卡？"，然后测量那个操作的延迟。系统内部指标（fps、write 耗时）只是辅助诊断手段，不是验收标准。

### 2. 压力测试强度要匹配真实负载，不能点到为止
- **问题**：最初只跑了 30 秒、25000 行的压力测试，远低于 Claude Code 实际运行时的持续高吞吐场景（几十分钟不间断输出）。
- **原因**：想快速得到结果，没有认真考虑真实场景的负载特征。Claude Code 同时在多个终端持续输出，吞吐量远超简单的 for 循环。
- **教训**：压力测试至少要运行 10-20 分钟以上，使用无限循环（`while ($true)`）模拟持续输出。短时间测试只能发现启动期问题，无法暴露内存积累、GC 压力、资源泄漏等长时间运行才会出现的问题。

### 3. 不要用过于间接/脆弱的方式做端到端测试
- **问题**：试图通过修改 app-state.json、API 调用、后台进程等"高级技巧"来自动化测试，结果反复遇到文件锁定、进程管理、ID 格式（`\r` 问题）等各种工程问题，浪费大量时间。
- **原因**：追求全自动化，忽略了当前基建水平。API Server 没有创建终端、切换布局的接口，强行通过修改状态文件来实现是脆弱的。
- **教训**：优先使用最直接的方式——让用户手动操作 UI（点击 6 分屏按钮），然后通过 API 发送命令。不要试图绕过 UI 自动化所有步骤。如果某个操作没有 API 支持，就承认需要人工介入，而不是用 hack 方式实现。

### 4. 修改 useEffect 时必须考虑 cleanup 函数对初始化流程的影响
- **问题**：添加 `setVisibility` 的 useEffect 时，cleanup 函数在 `visibleTerminals` 变化时（1→6 个终端）把旧终端设为 `visible=false`，触发 WebGL 上下文 dispose。紧接着新 effect 又设为 `visible=true` 重建 WebGL。这个一帧内的 dispose+create 导致 6 分屏初始化卡死。
- **原因**：没有充分考虑 React useEffect 的 cleanup 执行时机——依赖数组变化时 cleanup 在新 effect 之前执行，对于会产生副作用（WebGL 上下文操作）的 cleanup，这个时序是致命的。
- **教训**：涉及 GPU 资源（WebGL）、DOM 操作等有副作用的 useEffect，cleanup 函数要格外谨慎。如果 cleanup 会释放资源而新 effect 会重建，考虑是否需要 cleanup——很可能不需要，因为 effect 重新运行时会自动覆盖旧状态。

### 5. 性能优化要从数据流的源头开始，而不是只在末端打补丁
- **问题**：第一轮优化只在渲染器侧做了写入节流（50ms），writes/s 从 18000 降到 96，但用户输入延迟几乎没有改善（2212ms vs 原来的 1900ms）。
- **原因**：渲染器侧节流只减少了 `terminal.write()` 调用，但主进程仍在以 16ms 间隔向每个终端发送 IPC 消息（360 IPC/s）。渲染器主线程仍要处理每条 IPC 回调（查 Map、push 数据、检查调度），这本身就消耗了大量 CPU。
- **教训**：性能优化要沿着数据流从源头到末端逐级分析。PTY → TerminalDataBuffer(main) → IPC → renderer callback → pendingData → rAF → terminal.write()。如果只在最后一步优化，上游的开销仍然存在。应该同时优化 IPC 发送频率（主进程 16ms→100ms）和渲染器侧写入频率。

### 6. 进程管理必须使用精确名称，禁止通配符
- **问题**：使用 `taskkill` 和 `Get-Process` 时用通配符匹配进程名，导致可能误杀用户的其他进程。
- **原因**：图方便，没有考虑同时运行多个版本的 Onward 实例的情况。
- **教训**：已写入 CLAUDE.md 作为硬性规则：使用 `taskkill`、`Get-Process`、`pkill` 时必须用精确进程名，禁止通配符或部分匹配。Windows 上通过 `wmic process where "name='精确名称.exe'"` 来查询和操作。

### 7. Windows 环境下 bash 变量和 PowerShell 命令的转义陷阱
- **问题**：通过 curl 向 API 发送 PowerShell 命令时，bash 将 `$i`、`$_` 等 PowerShell 变量解释为 bash 变量（空值），导致 5/6 终端命令发送失败。同时 Python 输出的终端 ID 包含 `\r`（Windows 回车符），导致 curl URL 拼接错误。
- **原因**：跨 shell 调用（bash → curl → JSON → PowerShell）的转义层次复杂，每层都可能吞掉或改变特殊字符。
- **教训**：跨 shell 传递命令时，用 heredoc（`<< 'EOF'`）或文件方式传递 payload，避免多层转义。处理外部命令输出时始终 `tr -d '\r'` 清理 Windows 换行符。
